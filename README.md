# Lenovo Flash XML Helper

This folder contains an original helper workflow that decrypts Lenovo `.x`
firmware metadata files into plain XML, validates them against the firmware
package contents, and prepares files that SP Flash Tool can read.

This helper was built and tested locally against the firmware package:

- Device package: `TB351FU_ROW_OPEN_USER_M15125.2_A16_ZUI_17.5.10.073_ST_260213`
- Project from decrypted `flash.xml`: `t808aa`
- Visible model code in package name: `TB351FU`

This README is written specifically for the helper in this folder. It does not
copy vendor documentation.

## Files In This Helper Folder

- `install.sh`
  Creates a local Python virtual environment in `.venv` and installs the only
  required dependency.

- `run_lenovo_decrypt.py`
  Decrypts Lenovo `.x` metadata files and validates the resulting XML.

- `README.md`
  This guide.

- `.venv/`
  Created by the installer. Local runtime for the helper.

- `output/`
  Generated by the helper. Holds decrypted XML files and the validation report.

## What The Helper Does

- Decrypts `flash.x` into `flash.xml`
- Decrypts the scatter `.x` file referenced by `flash.xml`
- Decrypts efuse variants when present
- Verifies the XML files are well-formed
- Verifies `flash.xml` points to an existing DA and scatter file
- Verifies scatter partition entries contain valid hex addresses and sizes
- Verifies each referenced image file exists in the package
- Writes a machine-readable validation report as `validation_report.json`

## Installation

From this folder:

```bash
bash ./install.sh
```

What this does:

- creates `.venv`
- upgrades `pip` inside the virtual environment
- installs `pycryptodome`
- leaves your system Python untouched

## Basic Usage

Run the helper like this:

```bash
./.venv/bin/python ./run_lenovo_decrypt.py \
  --package-dir /path/to/ROM_FOLDER \
  --output-dir /path/to/output_folder
```

Optional explicit password:

```bash
./.venv/bin/python ./run_lenovo_decrypt.py \
  --package-dir /path/to/ROM_FOLDER \
  --output-dir /path/to/output_folder \
  --password OSD
```

## If The Helper Cannot Find Your Package Files

The helper expects this structure inside `--package-dir`:

```text
ROM_FOLDER/
  image/
    flash.x
    flash.xsd
    *.x
    DA_BR.bin or another DA file
    image files such as boot.img, super.img, vbmeta.img
```

If the helper fails to find files:

- Make sure `--package-dir` points to the ROM folder itself, not its parent.
- Make sure the ROM folder contains an `image` directory.
- Make sure `image/flash.x` exists.
- If your package uses a different DA filename or scatter filename, the helper
  will read that from decrypted `flash.xml`. The referenced file must still
  exist in the `image` directory.

Common path mistake:

- Wrong:
  `/some/path/new_updated`
- Right:
  `/some/path/new_updated/TB351FU_ROW_OPEN_USER_M15125.2_A16_ZUI_17.5.10.073_ST_260213`

That bundle is a single staging folder meant for SP Flash Tool. It contains:

- efuse XML files when present
- the DA file
- every image file referenced by the scatter

## How To Use The Generated Files In SP Flash Tool

The recommended file to load is the staged `flash.xml` inside
`sp_flash_tool_bundle/`.

Recommended preparation:

1. Use the generated staging folder, not the original root-level `flash.xml`.
2. Load `sp_flash_tool_bundle/flash.xml`.
3. Keep all files in `sp_flash_tool_bundle/` together.

For the package tested here, use:

- Flash XML:
  `lenovo_flash_xml_helper/output/TB351FU_ROW_OPEN_USER_M15125.2_A16_ZUI_17.5.10.073_ST_260213/sp_flash_tool_bundle/flash.xml`

### Linux SP Flash Tool Example

From the workspace root:

```bash
LD_LIBRARY_PATH=$PWD/SP_Flash_Tool_v6.2228_Linux:$PWD/SP_Flash_Tool_v6.2228_Linux/lib \
./SP_Flash_Tool_v6.2228_Linux/SPFlashToolV6 \
  -f "$HOME/Desktop/new_updated/TB351FU_ROW_OPEN_USER_M15125.2_A16_ZUI_17.5.10.073_ST_260213/lenovo_flash_xml_helper/output/TB351FU_ROW_OPEN_USER_M15125.2_A16_ZUI_17.5.10.073_ST_260213/sp_flash_tool_bundle/flash.xml" \
  -c download
```

### Windows SP Flash Tool Style

If you use the Windows tool, load the generated `flash.xml` from
`sp_flash_tool_bundle/`. The helper already copies `flash.xsd` and the
companion files there automatically.

## Practical Flashing Notes

- Double-check that the firmware matches the exact device model before flashing.
- Use the generated `flash.xml`, not the root-level package `flash.xml` that was
  found to be inconsistent for this ROM.
- Do not mix files from different firmware packages.
- Keep battery sufficiently charged.
- Use a stable USB cable and direct USB port.
- If SP Flash Tool reports missing schema or invalid XML, make sure you are
  loading the staged `flash.xml` from `sp_flash_tool_bundle/`.

## Troubleshooting

If install fails:

- Run `bash ./install.sh`
- Confirm `python3 -m venv` works on your system

If decrypt fails:

- Confirm `--package-dir` points to the folder that contains `image/`
- Confirm `image/flash.x` exists
- Try passing `--password OSD`

If SP Flash Tool says the XML is invalid:

- Confirm you loaded `sp_flash_tool_bundle/flash.xml`
- Confirm `sp_flash_tool_bundle/flash.xsd` exists
- Confirm the DA file exists in the same staging folder
- Confirm the required image files were copied into the same staging folder

If you want to review validation details:

- Open `validation_report.json`
